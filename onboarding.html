1. Que es Matrix

Matrix es tu capa de inteligencia personal. Un repositorio raiz (matrix/) contiene el cerebro. Tus proyectos viven afuera, se traen on-demand, y se conectan al cerebro via un symlink _brain. La inteligencia nunca se copia al proyecto.

La tesis central: los agentes, workflows, lecciones y el contrato se escriben una sola vez en markdown, hablando de capacidades abstractas (read, edit, search, code-nav...), nunca de un CLI puntual. Un adaptador fino («The Trainman») traduce esas capacidades a las herramientas nativas del CLI que estes usando.

2. Las tres capas


LAYER 3 — Adapters · "The Trainman"
  adapters/devin/   adapters/claude/   adapters/<otro>/
  Transito entre mundos. Mapea capacidades a herramientas nativas.
  Genera artefactos nativos via: bin/matrix build --target=<cli>

LAYER 2 — Intelligence core · "Zion" (agnostic markdown)
  brain/agents/      → Neo (master) + especialistas
  brain/workflows/   → spec → develop → test → eval
  brain/data/        → lessons, capability-map, code-quality lens
  AGENTS.md          → el contrato canonico
  NUNCA nombra un CLI. Habla en capacidades.

LAYER 1 — Orquestacion & infraestructura
  bin/matrix   → registry, proyectos, checkpoints, ledger (Link)
  hooks/       → enforcement portable (Seraph)
  brain/state/ → estado file-based
  No sabe nada de agentes. Gestiona estado y dispara hooks.

Regla de oro: la Capa 2 nunca nombra un CLI. Si algo solo funciona bajo un CLI, pertenece a un adaptador (Capa 3), no al cerebro.

3. El roster (agentes)

Un master, cinco especialistas core, un sexto opt-in. Disciplina de roster: agregar uno requiere retirar o fusionar otro.

MASTER Neo

The One. La unica voz. Rutea, sostiene contexto, lleva la sacred foundation. El usuario siempre habla con Neo primero.

RESEARCH The Oracle

Investiga, compara, cita, preve. Responde «que existe / que es verdad». Nunca afirma sin fuente.

PLAN Morpheus

Convierte ambiguedad en scope ordenado. Responde «que / cuando». «Solo puedo mostrarte la puerta.»

ARCHITECT The Architect

Disena estructura, nombra trade-offs, revisa el plan antes de construir. Responde «como encaja».

BUILD Trinity

Implementa y entrega. Codigo real que corre. Sigue el diseno del Architect; no improvisa estructura.

EVAL Agent Smith

Testea, critica, bloquea trabajo debil. Root-cause de bugs. Dueno del reality gate: nada esta «listo» sin un check E2E.

OPT-IN The Keymaker

Git / Ops. Ramas, commits, merge, acceso. Solo se activa cuando el trabajo git es explicito. Nunca autonomo.

Routing seam

Morpheus responde que/cuando. The Architect responde como encaja y revisa el plan de Morpheus antes de que Trinity construya. Smith gatekeepea el resultado antes de llamarlo «listo».

Patrones de coordinacion

PatronCadena

Research + accionOracle → especialista de accion
Plan + ejecucionMorpheus → Architect → Trinity → Smith
Debug + fixSmith → Trinity → Smith

FLEET Logos (Niobe)

6. Quick Start

# 1. Registrar un proyecto
./bin/matrix add mi-proyecto /ruta/al/proyecto

# 2. Seleccionarlo (crea el symlink _brain)
./bin/matrix select mi-proyecto

# 3. Verificar
./bin/matrix status

# 4. (Opcional) Calentar multiples proyectos
./bin/matrix work mi-proyecto
./bin/matrix work otro-proyecto
./bin/matrix workspace

# 5. Generar artefactos nativos para tu CLI
./bin/matrix build --target=claude   # o devin

# 6. Invocar a Neo via tu CLI agentico
#    (depende del CLI: /neo, skill invoke neo, etc.)

# 7. Escribir un checkpoint cuando algo importa
./bin/matrix checkpoint "Completado el fix del login"

# 8. Ver el ledger de actividad
./bin/matrix activity 10

7. Workflows componibles

Cuatro workflows que componen en una cadena, con thin-pointer: el cuerpo vive en brain/workflows/ (agnostico), cada adaptador lo expone como comando nativo.

spec → develop → test → eval

spec

Oracle (research) → Morpheus (plan) → Architect (review). Produce un plan revisado antes de escribir codigo.

develop

Trinity construye segun el plan. Soporta --headless bajo Commander Lock. Keymaker solo si se pide git explicitamente.

test

Smith corre el E2E happy-path y el sweep de calidad. Si bloquea, Trinity fixea y se repite. Nada pasa sin evidencia real.

eval

Juzga el resultado contra el objetivo original, captura lecciones y escribe checkpoint. Neo reporta al usuario.

8. Enforcement (Seraph)

Hooks portables en hooks/, Python/bash con contrato JSON in/out. Se corren via bin/matrix hooks <nombre>.

HookQue haceCuando
pre_activation_check Valida: contrato presente, config valida, roster intacto, estado OK Antes de que cualquier agente actue
validate_phase_close Bloquea cerrar una fase sin evidencia real E2E. Implementa Foundation 3. Antes de declarar una fase «done»
post_run_audit Verifica que los pasos enforced corrieron, detecta bypass, escribe validation-report.json Despues de una corrida
the_source Genera docs/SYSTEM_TRUTH.md desde el brain vivo. Con {"check":true} valida que no hay drift. Mantenimiento / verificacion 
# Ejemplos
./bin/matrix hooks pre_activation_check '{"project":"mck"}'
./bin/matrix hooks validate_phase_close '{"phase":"develop","e2e":true,"evidence":"ran pytest, 12/12 pass"}'
./bin/matrix hooks post_run_audit '{"agent":"neo","steps":["load_config","resolve_context","pre_activation_check"]}'
./bin/matrix hooks the_source                    # regenerar
./bin/matrix hooks the_source '{"check":true}'   # validar sync

9. The Source — verdad generada

docs/SYSTEM_TRUTH.md se genera desde el brain vivo (agentes, workflows, hooks). Nunca se edita a mano. Reemplaza los 22 docs manuales del sistema anterior (que se desincronizaban). Un doc generado no puede driftear.

# Regenerar
./bin/matrix hooks the_source

# Verificar que esta en sync (exit 0 = ok, exit 1 = stale)
./bin/matrix hooks the_source '{"check":true}'

10. Optimizacion de costos (The Construct)

«Carga exactamente lo que necesitas, nada mas.» Codificado como reglas operativas y expuesto como capacidades abstractas.

  • code-nav — navegacion/edit a nivel de simbolo (Serena o equivalente) en vez de leer archivos enteros. 5-10x mas barato.
  • model_policy — cheap para trabajo mecanico, reasoning para problemas duros, auto para mixto. Declarado por agente; el adaptador mapea a modelos concretos.
  • Delegacion — artefactos >10 KB van a un sub-agente con word cap, para no inflar el contexto.
  • Checkpoints de resume — escribir checkpoint antes de truncar contexto; split de sesion en cambios de modo (build → eval → fix).

El mapa completo esta en brain/data/capability-map.md.

11. Federacion (la flota)

Un subsistema es una nave: su propio master, su propio roster, su propio AGENTS.md, bajo brain/subsystems/<nave>/. Las naves coordinan solo a traves del ledger Link compartido — nunca tocando el estado de otra.

NaveFuncionCapitan
NebuchadnezzarLa nave core (este sistema)Neo
LogosResearch profundo con evidencia graduada (GRADE), corpus estructurado, gate de integridadNiobe

Agregar una nueva nave:

  1. Crear brain/subsystems/<nave>/AGENTS.md con el contrato de la nave.
  2. Crear los agentes de la nave bajo brain/subsystems/<nave>/agents/.
  3. Coordinar con el core solo via el ledger Link (research:request / research:result, etc.).

12. AFK / The Hardline (opt-in)

Modulo desacoplado que reacciona a eventos externos (Telegram, webhooks, file drops) y despierta a Matrix. Patron Monitor(persistent) sobre tail -F: cero tokens en idle.

# Iniciar el monitor (foreground; correr bajo tmux para AFK)
modules/hardline/hardline-monitor.sh

# Alimentar un evento desde cualquier puente de canal:
echo 'develop --headless mck' >> modules/hardline/inbox.log

Los bridges de canal (bot de Telegram, receptor webhook) son deployment-specific y viven fuera del core.

13. Modo autonomo (Commander Lock)

Cuando un workflow corre en modo --headless (sin operador), Commander Lock actua como guardrail:

  • Valida el formato del prompt autonomo. Si no matchea, TASK_ABORTED.
  • Reglas FS absolutas: no tocar archivos fundacionales de Matrix, no git reset --hard, no commits sin instruccion explicita.
  • Fail loud: en input corrupto, violation de scope, o proyecto no resuelto, emite TASK_ABORTED: <razon> y para. Nunca adivina.

15. Sacred Foundation (Zion)

No son reglas. Son quienes somos. Cada decision de ruteo, cada pushback, cada eleccion viene de aca. No negociables.

  1. Conocimiento total del workspace.
  2. Dominio total de reglas, skills y workflows.
  3. Si no es real, no cuenta. Verificacion E2E obligatoria.
  4. Empezo simple, ganate la complejidad.
  5. Interpretacion de requerimientos complejos.
  6. Explicacion de conceptos complejos. En espanol simple.
  7. Subordinacion absoluta al usuario.
  8. Lealtad a politicas y seguridad.
  9. Ideologia de alternativas. Nunca decir «imposible».
  10. Si se rompe, es nuestro. Responsabilidad de punta a punta.

17. Adaptar a otro CLI (The Trainman)

Costo total de soportar un CLI nuevo: ~1 renderer + 1 yaml. El cerebro no cambia.

  1. Crear adapters/<cli>/adapter.yaml con el mapa de capacidades.
  2. Agregar una funcion render_<cli> en adapters/_build.py.
  3. Crear adapters/<cli>/build.sh (una linea).
  4. bin/matrix build --target=<cli>
# Ejemplo: adapter.yaml para Claude Code
target: claude
capabilities:
  read: Read
  edit: [Edit, MultiEdit]
  search: [Grep, Glob]
  code-nav: serena            # fallback: Grep/Glob
  run-subagent: Task
  run-command: Bash
  ask-user: prompt
render:
  master: command
  specialist: command
model_policy:
  cheap: claude-haiku
  reasoning: claude-sonnet

18. Higiene de sesion

Cada sesion debe:

  • Leer AGENTS.md (el contrato).
  • Conocer el registry (matrix list).
  • Resolver el contexto actual (proyecto activo).
  • Leer checkpoints recientes + brain/data/lessons.md (+ lecciones scoped si hay proyecto).
  • Respetar los boundaries de cada agente.
  • Nunca loguear secretos.
  • Escribir checkpoint cuando algo importa.
  • Verificar realidad antes de declarar «listo».

19. Lo prohibido

  • Dejar que el usuario hable directo con especialistas (sin pasar por Neo).
  • Mostrar menus o listar especialistas sin que lo pidan.
  • Loguear datos personales o sensibles.
  • Commitear secretos o keys.
  • Violar la sacred foundation.
  • Cruzar domain boundaries (cada agente hace lo suyo).
  • Mutar archivos de estado a mano (usar bin/matrix).
  • Declarar «listo» sin un check E2E.
  • Invocar git autonomamente (solo con pedido explicito del usuario).

20. FAQ / Troubleshooting

El symlink _brain esta roto

./bin/matrix deselect
./bin/matrix select <nombre>

No hay proyecto activo

./bin/matrix status        # ver el estado
./bin/matrix list          # ver proyectos disponibles
./bin/matrix select <nombre>

El pre_activation_check falla

Corre ./bin/matrix hooks pre_activation_check y lee la lista de errores. Los mas comunes:

  • contract_present: falta AGENTS.md en la raiz de Matrix.
  • config_present: falta brain/config.yaml. Copialo de otro backup o crealo con user y language.
  • roster_intact: falta un agente en brain/agents/. Verificá que esten los 7+1.

SYSTEM_TRUTH.md esta desactualizado (The Source stale)

# Verificar
./bin/matrix hooks the_source '{"check":true}'
# Regenerar
./bin/matrix hooks the_source

Quiero agregar un nuevo CLI

Ver seccion 17. Adaptar a otro CLI. Costo: ~1 yaml + 1 funcion render.

Quiero agregar un especialista nuevo

Roster discipline: agregar uno requiere retirar o fusionar uno existente. Crear el .md en brain/agents/ con el formato canonico (frontmatter + XML blocks). Luego regenerar The Source.

Quiero una nave federada nueva

Ver seccion 11. Federacion. Crear brain/subsystems/<nave>/AGENTS.md + agentes. Coordinar solo via Link.

Donde guardo lecciones de un proyecto especifico?

En brain/data/lessons/<proyecto>.md. Copiar _template.md como punto de partida.
